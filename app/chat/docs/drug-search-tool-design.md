# Drug Search Tool Design

## 목적

이 문서는 KADA 도핑 챗봇에서 약물 검색 기능을 어떻게 설계할지 정리한다.

챗봇의 최종 목표는 사용자가 KADA 홈페이지에 직접 들어가지 않아도, 제품명 또는 성분명을 입력해 도핑 관련 위험을 빠르게 확인할 수 있게 하는 것이다.

다만 약물 사용 가능 여부는 제품명, 성분명, 경기기간 여부, 투여 경로, 종목, 용량, TUE 가능성에 따라 달라질 수 있으므로 챗봇이 단정적으로 "사용 가능"이라고 답하지 않도록 설계한다.

## 설계 원칙

```text
1. 사용자 입력은 자연어로 받을 수 있게 한다.
2. 판단에 필요한 정보가 부족하면 추가 질문을 한다.
3. 제품명과 성분명을 모두 검색 대상으로 삼는다.
4. 검색 결과가 여러 개면 사용자가 정확한 제품을 선택하게 한다.
5. 제품이 불명확하면 성분명, 제품 사진, 포장 정보 확인을 요청한다.
6. 최종 답변은 강한 사용 가능/불가능 단정이 아니라 위험 수준으로 표현한다.
7. 공식 검색 결과와 규정 문서 근거를 함께 제시한다.
```

## 입력 방식

약물 질문은 혼합형으로 처리한다.

사용자는 처음에 자유롭게 질문할 수 있다.

예시:

```text
타이레놀 먹어도 돼?
코감기 스프레이 써도 되나요?
아세트아미노펜은 경기기간 중에 괜찮나요?
```

챗봇은 입력에서 제품명, 성분명, 경기기간 여부를 먼저 추출한다.

정보가 부족하거나 판단에 영향을 줄 가능성이 있으면 추가 질문을 한다.

## 기본 확인 항목

우선 확인해야 하는 항목은 다음과 같다.

| 항목 | 필수 여부 | 설명 |
|---|---|---|
| 제품명 | 기본 확인 | 사용자가 가장 자주 입력하는 정보 |
| 성분명 | 기본 확인 | 실제 도핑 판단에 중요한 정보 |
| 경기기간 여부 | 기본 확인 | 경기기간 중 금지와 상시 금지를 구분하기 위해 필요 |
| 투여 경로 | 조건부 확인 | 투여 경로에 따라 금지 여부가 달라지는 경우 질문 |
| 종목 | 조건부 확인 | 특정 종목에서만 금지되거나 제한이 있는 경우 질문 |
| 용량/사용량 | 조건부 확인 | 임계값 또는 용량 기준이 있는 경우 질문 |
| TUE 여부 | 조건부 확인 | 치료 목적 사용면책이 필요할 가능성이 있는 경우 질문 |

## 조건부 추가 질문

처음부터 모든 항목을 요구하지 않는다.

투여 경로와 종목은 판단에 영향을 줄 가능성이 있을 때만 표출한다.

예시:

```text
이 성분은 투여 경로에 따라 판단이 달라질 수 있습니다.
경구 복용, 주사, 흡입, 비강 분사, 피부 도포 중 어떤 방식인가요?
```

```text
일부 약물군은 특정 종목에서만 금지될 수 있습니다.
해당 선수의 종목을 알려주세요.
```

## 검색 대상

검색은 제품명과 성분명을 모두 허용한다.

| 입력 유형 | 처리 방식 |
|---|---|
| 제품명 | KADA 약물검색 또는 약품 데이터에서 후보 제품 검색 |
| 성분명 | 금지목록 분류, 경기기간 금지 여부, 상시 금지 여부와 연결 |
| 제품명 + 성분명 | 제품 후보와 성분 기준을 함께 확인 |
| 모호한 표현 | 추가 질문으로 제품명 또는 성분명 확인 |

제품명으로 들어온 질문은 성분명 확인을 유도한다.

성분명으로 들어온 질문은 WADA 금지목록, KADA 금지약물 검색서비스, KADA FAQ와 연결한다.

## 검색 결과 후보 처리

검색 결과가 여러 개일 수 있으므로 챗봇이 임의로 하나를 확정하지 않는다.

처리 방식:

```text
1. 상위 후보 제품을 보여준다.
2. 사용자가 정확한 제품을 선택하게 한다.
3. 후보가 애매하면 성분명, 제품 사진, 포장 정보 확인을 요청한다.
4. 제품이 확정되기 전에는 복용 가능 여부를 단정하지 않는다.
```

예시 답변:

```text
검색 결과가 여러 개입니다.
정확한 제품을 선택해 주세요.

1. 타이레놀정 500mg
2. 타이레놀 8시간 이알서방정
3. 어린이 타이레놀 현탁액

제품 포장 또는 성분표에 적힌 성분명도 함께 확인하면 더 안전합니다.
```

## 위험 상태값

최종 표현은 사용 가능/사용 불가처럼 강하게 단정하지 않고 위험 수준으로 안내한다.

| 상태값 | 의미 |
|---|---|
| 낮은 위험 | 현재 입력 정보 기준 금지 가능성이 낮아 보이는 경우 |
| 주의 필요 | 경기기간, 투여 경로, 종목, 용량 등에 따라 판단이 달라질 수 있는 경우 |
| 금지 가능성 있음 | 금지목록 또는 약물검색에서 관련 성분이나 분류가 확인되는 경우 |
| 확인 필요 | 제품명, 성분명, 투여 경로, 검색 결과가 불명확한 경우 |

`낮은 위험`도 사용 가능을 보장하는 표현이 아니다.

답변에는 항상 확인 기준과 검색 기준일을 함께 표시한다.

## 답변 구조

약물 검색 답변은 다음 구조를 따른다.

```text
1. 현재 입력 정보
2. 확인된 핵심 정보
3. 추가 확인이 필요한 정보
4. 권장 행동
5. 근거
```

예시:

```text
현재 입력 정보
- 제품명: 타이레놀
- 성분명: 아세트아미노펜
- 경기기간 여부: 경기기간 중

확인된 핵심 정보
- 현재 입력 기준 핵심 확인 성분은 아세트아미노펜입니다.
- 제품명이 같아도 제형과 복합 성분에 따라 결과가 달라질 수 있습니다.

추가 확인이 필요한 정보
- 정확한 제품명 또는 성분표 확인이 필요합니다.
- 복합 감기약이면 다른 성분이 포함되어 있을 수 있습니다.

권장 행동
- KADA 약물검색 기준으로 제품명과 성분명을 함께 확인합니다.
- 경기기간 중이면 복용 전 팀 닥터, 약사, 도핑 담당자에게 확인합니다.

근거
- KADA 금지약물 검색서비스
- WADA 금지목록 국제표준
- 검색 기준일: YYYY-MM-DD
```

## RAG와 Tool의 역할 분리

RAG는 규정, 절차, 판단 기준을 설명하는 데 강하다.

하지만 특정 제품의 현재 금지 여부를 판단하려면 최신 약물 데이터와 검색 결과가 필요하다.

따라서 역할을 다음처럼 나눈다.

| 구성요소 | 역할 |
|---|---|
| RAG retriever | 금지목록, TUE, 검사 절차, 선수 권리와 의무 검색 |
| Drug search tool | 제품명/성분명 기반 약물 검색 |
| Answer generator | RAG 근거와 약물검색 결과를 합쳐 답변 생성 |
| Guardrail | 단정 표현 방지, 추가 확인 질문, 공식 근거 표시 |

## 현재 단계의 한계

현재 구현은 규정 문서 RAG와 manual source를 기반으로 약물 확인 절차를 안내한다.

아직 KADA 약물검색을 자동으로 조회하지 않기 때문에 특정 제품에 대해 최종 위험 상태를 확정할 수 없다.

현재 단계에서 가능한 답변:

```text
제품명과 성분명, 경기기간 여부를 확인해야 합니다.
투여 경로와 종목에 따라 판단이 달라질 수 있습니다.
KADA 약물검색 또는 공식 근거 확인이 필요합니다.
```

아직 하면 안 되는 답변:

```text
이 제품은 무조건 사용 가능합니다.
이 제품은 무조건 금지입니다.
스프레이 형태라면 항상 괜찮습니다.
```

## 다음 구현 단계

1. KADA 약물검색 페이지의 검색 방식 조사
2. 자동 조회 가능성 검토
3. 제품 후보 결과 schema 정의
4. 위험 상태값 산출 로직 정의
5. retrieval answer prompt에 drug search result 연결
6. Gradio UI에서 약물 질문 전용 입력 흐름 구현
7. 이후 MCP tool로 감싸 외부 agent가 호출할 수 있게 확장

## MCP Tool 확장 방향

향후 MCP tool로 감쌀 때는 다음 입력과 출력을 기준으로 설계한다.

입력:

```json
{
  "query": "타이레놀",
  "ingredient": "아세트아미노펜",
  "in_competition": true,
  "route": null,
  "sport": null
}
```

출력:

```json
{
  "status": "확인 필요",
  "matched_products": [],
  "matched_substances": [],
  "requires_route_confirmation": false,
  "requires_sport_confirmation": false,
  "recommended_action": "KADA 약물검색 결과 확인 후 복용 여부를 결정하세요.",
  "sources": []
}
```

## 발표 포인트

이 설계의 핵심은 약물 질문을 일반 RAG 질문으로만 처리하지 않는 것이다.

제품명 기반 질문은 최신성, 정확성, 투여 경로, 종목 조건이 중요하기 때문에 RAG와 별도 tool을 분리했다.

RAG는 규정과 절차를 설명하고, drug search tool은 제품명과 성분명 조회를 담당한다.

이 구조는 이후 LangGraph에서 다음과 같은 분기 흐름으로 확장할 수 있다.

```text
사용자 질문
-> 약물 질문인지 분류
-> 제품명/성분명 추출
-> 정보 부족 시 추가 질문
-> drug search tool 호출
-> RAG로 관련 규정 검색
-> 위험 상태값과 근거 기반 답변 생성
```
